## 1. 문제 정의

중견·대기업 내부 팀은 매일 수십 건의 회의를 진행하지만, 회의 후 공식 회의록을 작성하는 데 평균 30~60분을 소비하고, 담당자별 후속 액션 아이템이 산발적으로 관리되어 누락·지연이 빈번하다. 기존 협업 툴(Slack, Teams 등)은 실시간 소통은 지원하지만 회의 내용을 자동으로 요약·구조화하거나, 참여자별 할 일 목록을 생성하는 기능이 부재하다. 이 문제를 해결하지 않으면 회의 후속 관리의 비효율이 지속되어 프로젝트 지연과 책임 소재 불명확이 반복된다.

---

## 2. 목표

**사용자 목표**

- 회의 종료 후 5분 이내에 AI가 생성한 회의록 초안을 확인할 수 있다.
- 회의 참여자별로 자동 생성된 투두리스트를 개인 대시보드에서 즉시 확인하고 관리할 수 있다.
- 실시간 채팅을 통해 회의 전·후 맥락을 한 채널에서 연속적으로 공유할 수 있다.

**비즈니스 목표**

- 출시 후 30일 이내 팀당 주간 활성 사용자(WAU) 70% 이상 달성한다.
- 회의록 작성에 소요되는 시간을 팀 평균 70% 이상 단축시킨다.
- 해커톤 발표 기준: 2시간 내 핵심 기능(녹음 업로드 → 회의록 + 투두) 동작 데모 완성.

## 3. 비목표 (Non-Goals)

| 비목표 | 제외 이유 |
|---|---|
| 영상 통화·화상 회의 기능 | 별도의 복잡한 인프라(WebRTC 등) 필요, v1 범위 초과 |
| 다국어 동시통역 | 음성 처리 정확도가 MVP 가치보다 우선되므로 단일 언어(한국어)로 제한 |
| 외부 캘린더(Google, Outlook) 자동 연동 | OAuth + 캘린더 API 연동은 v2 이후 |
| Zoom·Google Meet 자동 녹음 수집 | 외부 API 의존성 높음, 해커톤 내 구현 불가 |
| 회의록 전자 결재·승인 워크플로 | 기업 내 결재 시스템 연동 필요, 별도 프로젝트로 분리 |

## 4. 사용자 스토리

### 페르소나 A: 팀 리더 / 회의 주재자

- **A-1.** 팀 리더로서, 회의 후 로컬 녹음 파일(.mp3/.mp4/.wav)을 업로드하면 AI가 자동으로 회의록 초안을 생성해 주길 원한다. 그래야 회의 직후 바로 팀원들과 내용을 공유할 수 있기 때문이다.
- **A-2.** 팀 리더로서, AI가 생성한 회의록에서 참여자별 액션 아이템이 자동으로 분류되어 투두리스트로 제공되길 원한다. 그래야 "누가 무엇을 해야 하는지"를 별도로 정리하지 않아도 되기 때문이다.
- **A-3.** 팀 리더로서, 채팅 채널에 회의록 링크를 공유하고 팀원들과 실시간으로 피드백을 주고받을 수 있길 원한다. 그래야 회의 후속 논의가 이메일 대신 한 곳에서 이루어지기 때문이다.

### 페르소나 B: 팀원 / 회의 참여자

- **B-1.** 팀원으로서, 내 이름이 태깅된 투두리스트를 대시보드에서 확인하고 완료 처리할 수 있길 원한다. 그래야 "내가 맡은 일이 무엇인지"를 놓치지 않을 수 있기 때문이다.
- **B-2.** 팀원으로서, 회의에 참석하지 못했을 때 회의록을 빠르게 읽고 관련 채팅 스레드를 통해 질문을 남길 수 있길 원한다. 그래야 회의에 없었어도 맥락을 놓치지 않기 때문이다.
- **B-3.** 팀원으로서, 내 투두 항목의 마감일을 직접 설정하고 완료 여부를 표시할 수 있길 원한다. 그래야 내 일정에 맞게 작업을 관리할 수 있기 때문이다.

### 페르소나 C: 관리자 / IT 담당자

- **C-1.** 관리자로서, 팀 내 채널과 멤버를 초대·관리할 수 있길 원한다. 그래야 조직 구조에 맞게 협업 공간을 분리할 수 있기 때문이다.

## 5. 기능 요구사항

### P0 — Must Have (해커톤 MVP)

#### 5.1 회의 녹음 업로드 및 AI 분석

| ID | 요구사항 | 인수 기준 |
|---|---|---|
| F-01 | 사용자는 .mp3, .mp4, .wav, .m4a 형식의 녹음 파일을 업로드할 수 있다 | - 파일 크기 최대 500MB 허용<br>- 업로드 중 프로그레스 바 표시<br>- 지원하지 않는 형식 업로드 시 명확한 오류 메시지 표시 |
| F-02 | 시스템은 업로드된 녹음 파일을 STT(Speech-to-Text)로 변환하고 AI가 회의록 초안을 생성한다 | - 변환 완료 후 자동으로 회의록 초안이 해당 채널에 게시됨<br>- 회의록에는 요약, 주요 논의 사항, 결정 사항이 포함됨<br>- 처리 중 사용자에게 상태 표시 (예: "AI가 회의록을 생성 중입니다...") |
| F-03 | AI는 회의록에서 참여자를 인식하고 각 참여자별 액션 아이템(투두)을 자동 추출한다 | - 발화자 이름 또는 언급된 이름 기준으로 투두 항목 분류<br>- 투두에는 내용, 담당자, 마감일(추정 또는 미정) 포함<br>- 투두 자동 생성 후 편집 가능 |

#### 5.2 참여자별 투두리스트

| ID | 요구사항 | 인수 기준 |
|---|---|---|
| F-04 | 각 사용자는 개인 투두 대시보드에서 자신에게 할당된 투두 항목을 확인할 수 있다 | - 로그인 후 메인 대시보드에 "내 할 일" 섹션 표시<br>- 어느 회의에서 생성된 투두인지 출처(회의명, 날짜) 표시 |
| F-05 | 사용자는 투두 항목을 완료 처리, 마감일 수정, 삭제할 수 있다 | - 체크박스로 완료 처리 시 시각적 완료 표시<br>- 완료된 투두는 "완료 목록"으로 이동 (삭제되지 않음)<br>- 마감일 수정 가능 |

#### 5.3 실시간 채팅

| ID | 요구사항 | 인수 기준 |
|---|---|---|
| F-06 | 사용자는 채널 내에서 실시간 텍스트 메시지를 주고받을 수 있다 | - 메시지 전송 즉시 채널 내 모든 참여자에게 표시 (WebSocket)<br>- 메시지에 텍스트, 이모지 지원 |
| F-07 | 채널 내에서 회의록과 투두리스트 링크를 공유할 수 있다 | - 회의록 생성 완료 시 채널에 자동으로 카드 형태로 공유<br>- 카드 클릭 시 상세 회의록 페이지로 이동 |

## 6. 성공 지표

### 단기 지표 (Leading Indicators) — 출시 후 1~4주

| 지표 | 목표 | 측정 방법 |
|---|---|---|
| 녹음 업로드 완료율 | 업로드 시도 대비 80% 이상 성공 | 업로드 API 성공/실패 로그 |
| 회의록 생성 성공률 | AI 처리 성공률 95% 이상 | STT + AI 파이프라인 에러율 |
| 투두 확인율 | 생성된 투두 중 72시간 내 확인 비율 70% 이상 | 투두 조회 이벤트 로그 |
| 채팅 메시지 전송수 / 일 | 팀당 평균 20건 이상 | 채팅 이벤트 로그 |
| 기능 활성화율 | 가입 후 7일 내 녹음 업로드 1회 이상 비율 60% | 사용자 행동 퍼널 |

### 장기 지표 (Lagging Indicators) — 출시 후 1~3개월

| 지표 | 목표 | 측정 방법 |
|---|---|---|
| 회의록 작성 시간 단축률 | 팀 평균 70% 단축 | 설문 또는 인터뷰 |
| 주간 활성 사용자(WAU) | 팀당 WAU 70% 이상 | 로그인 로그 |
| 투두 완료율 | 생성된 투두 중 마감 내 완료 비율 60% 이상 | 투두 상태 변경 로그 |
| 사용자 순추천지수(NPS) | 30점 이상 | 분기별 인앱 설문 |

## 8. 타임라인 및 구현 단계

### Phase 0: 해커톤 데모 (오늘, 2시간)

| 시간 | 목표 | 담당 |
|---|---|---|
| 0~30분 | 프로젝트 세팅 (Next.js/React + 백엔드 프레임워크), DB 스키마 설계 | 엔지니어링 |
| 30~60분 | 실시간 채팅 기본 구현 (WebSocket), 녹음 파일 업로드 UI 구현 | 엔지니어링 |
| 60~90분 | STT + AI 회의록 생성 파이프라인 연결 (Whisper API + GPT), 투두 자동 생성 로직 | 엔지니어링 |
| 90~120분 | 투두 대시보드 UI 완성, 통합 테스트 및 데모 준비 | 전체 팀 |

**해커톤 MVP 기술 스택 제안**

- **프론트엔드**: Next.js 14 (App Router) + Tailwind CSS
- **백엔드**: FastAPI (Python) 또는 Next.js API Routes
- **실시간 채팅**: Socket.io 또는 Supabase Realtime
- **STT**: OpenAI Whisper API
- **AI 회의록/투두 생성**: OpenAI GPT-4o (프롬프트 엔지니어링)
- **DB**: Supabase (PostgreSQL) 또는 Firebase

---

### Phase 1: MVP 안정화 (해커톤 이후 ~4주)

- F-08 인앱 녹음, F-09 회의록 편집, F-10 투두 재지정, F-11 파일 첨부

### Phase 2: 성장 (2~3개월 후)

- F-12 멘션/알림, F-13 캘린더 연동
- 성능 최적화 (대용량 파일 처리, 동시 접속자 처리)

### Phase 3: 확장 (3개월 이후)

- F-14~F-17 외부 툴 연동, 분석 대시보드, 다국어 지원

## 부록: 핵심 사용자 플로우

```
[사용자] 녹음 파일 업로드
    ↓
[시스템] STT 변환 (Whisper API)
    ↓
[시스템] AI 요약 + 참여자별 투두 추출 (GPT-4o)
    ↓
[채널] 회의록 카드 자동 게시 + 채팅 알림
    ↓
[각 참여자] 개인 대시보드에서 내 투두 확인 및 관리
```
